# Job Application Tracker — V2

Builds on V1. V1 must be complete and working before starting this phase.

V2 scope: add remaining CRUD endpoints, plug in an AI scoring pipeline
using OpenRouter, and persist the results.

| Area          | V1                        | V2                                      |
|---------------|---------------------------|-----------------------------------------|
| Endpoints     | POST /api/v1/jobs only    | Full CRUD + /analyze endpoint           |
| AI            | None                      | OpenRouter API (JD parser + scorer)     |
| Database      | jobs table only           | + agent_runs + scores tables            |
| Config        | DB credentials only       | + OPENROUTER_API_KEY env var            |

## New Database Migrations

### V2__create_agent_runs.sql

```sql
CREATE TABLE agent_runs (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id      UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    agent_name  VARCHAR(100) NOT NULL,
    model_used  VARCHAR(100),
    input_text  TEXT,
    output_text TEXT,
    status      VARCHAR(50) NOT NULL DEFAULT 'PENDING',
    created_at  TIMESTAMP NOT NULL DEFAULT now()
);
```

### V3__create_scores.sql

```sql
CREATE TABLE scores (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    job_id          UUID NOT NULL REFERENCES jobs(id) ON DELETE CASCADE,
    fit_score       INTEGER CHECK (fit_score BETWEEN 0 AND 100),
    recommendations TEXT,
    created_at      TIMESTAMP NOT NULL DEFAULT now()
);
```

## AI Scoring Pipeline

Triggered by `POST /api/v1/jobs/{id}/analyze`

All AI calls go through OpenRouter. OpenRouter is an API gateway that
routes to multiple model providers (Mistral, Llama, etc.) using a single
unified endpoint that is OpenAI-compatible.

```
POST /api/v1/jobs/{id}/analyze
        │
        ▼
OrchestratorService
        │
        ├── Step 1: JdParserAgent
        │     Input:  jd_text from jobs table
        │     Output: extracted skills, seniority level, domain
        │     Logs:   agent_runs (agent_name = 'JdParserAgent')
        │
        └── Step 2: ResumeScorerAgent
              Input:  parsed JD output + hardcoded resume summary
              Output: fit_score (0–100) + top 3 recommendations
              Logs:   agent_runs (agent_name = 'ResumeScorerAgent')
              Saves:  scores table + updates jobs.fit_score
```

## OpenRouter Integration

### API Details

- Base URL: `https://openrouter.ai/api/v1`
- Endpoint: `POST /chat/completions`
- Auth header: `Authorization: Bearer ${OPENROUTER_API_KEY}`
- Protocol: OpenAI-compatible (same request/response shape as OpenAI)
- Free tier models to use: `mistralai/mistral-7b-instruct` or `meta-llama/llama-3-8b-instruct`

### Required Headers

```
Authorization: Bearer ${OPENROUTER_API_KEY}
Content-Type: application/json
HTTP-Referer: http://localhost:8080
X-Title: Job Application Tracker
```

### Request Shape

```json
{
  "model": "mistralai/mistral-7b-instruct",
  "messages": [
    { "role": "system", "content": "system prompt here" },
    { "role": "user",   "content": "user prompt here" }
  ],
  "temperature": 0.3,
  "max_tokens": 500
}
```

### Response Shape

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "agent response text here"
      }
    }
  ]
}
```

Extract response with: `response.choices[0].message.content`

## New Classes to Add

```
src/main/java/com/jobtracker/
├── agents/
│   ├── OrchestratorService.java      ← coordinates agent calls
│   ├── JdParserAgent.java            ← parses job description
│   └── ResumeScorerAgent.java        ← scores resume fit
├── client/
│   └── OpenRouterClient.java         ← handles all HTTP calls to OpenRouter
├── model/
│   ├── AgentRun.java                 ← JPA entity
│   └── Score.java                    ← JPA entity
└── repository/
    ├── AgentRunRepository.java
    └── ScoreRepository.java
```

### OpenRouterClient responsibilities
- Single place for all OpenRouter HTTP calls
- Reads OPENROUTER_API_KEY from environment
- Accepts model name, system prompt, user prompt
- Returns raw response string
- Handles HTTP errors and throws descriptive exceptions
- Use Spring's RestClient (Spring Boot 3.2+) — no extra dependencies needed

### OrchestratorService responsibilities
- Only entry point to agent calls — nothing else calls OpenRouter directly
- Calls JdParserAgent then ResumeScorerAgent in sequence
- Logs every agent call to agent_runs table (input, output, status, model used)
- On any agent failure: log status = FAILED in agent_runs, throw exception
- Does not swallow errors silently

### JdParserAgent responsibilities
- System prompt: instructs model to extract skills, seniority, domain as JSON
- Returns structured output — parse the JSON from the model response
- Expected output shape:
```json
{
  "skills": ["Java", "Spring Boot", "PostgreSQL"],
  "seniority": "Senior",
  "domain": "Backend"
}
```

### ResumeScorerAgent responsibilities
- System prompt: instructs model to score resume fit and give recommendations
- Hardcode a short resume summary as a constant in this class for V2
- Expected output shape:
```json
{
  "fit_score": 82,
  "recommendations": [
    "Highlight GCP experience",
    "Add Kotlin to skills section",
    "Emphasize distributed systems work"
  ]
}
```

## New Environment Variable

| Variable           | Description               | Required |
|--------------------|---------------------------|----------|
| OPENROUTER_API_KEY | OpenRouter API key        | Yes      |

Add to local run:
```bash
export OPENROUTER_API_KEY=your_key_here
./gradlew bootRun
```

Add to `application.yml`:
```yaml
openrouter:
  api-key: ${OPENROUTER_API_KEY}
  base-url: https://openrouter.ai/api/v1
  model: mistralai/mistral-7b-instruct
```

## Coding Conventions (unchanged from V1)

- Controllers are thin — no business logic
- Business logic lives in `@Service` classes
- `OrchestratorService` is the only class that calls `OpenRouterClient`
- All tables use `snake_case`
- Every table has `id` (UUID), `created_at`
- Never hardcode API keys — use environment variables
- Never edit a Flyway migration after it has run — write a new one
- Base package: `com.jobtracker`

## Verify V2 is Working

```bash
# Start postgres
docker-compose up -d

# Run app
export OPENROUTER_API_KEY=your_key_here
./gradlew bootRun

# Insert a job with a real JD
curl -X POST http://localhost:8080/api/v1/jobs \
  -H "Content-Type: application/json" \
  -d '{
    "company": "Acme Corp",
    "role": "Senior Software Engineer",
    "jdText": "We are looking for a Senior Java engineer with Spring Boot, PostgreSQL, and GCP experience."
  }'

# Copy the returned id and trigger analysis
curl -X POST http://localhost:8080/api/v1/jobs/{id}/analyze

# Check the score
curl http://localhost:8080/api/v1/jobs/{id}/score

# Verify in database
docker exec -it job-tracker-db psql -U jobuser -d jobtracker -c "SELECT * FROM scores;"
docker exec -it job-tracker-db psql -U jobuser -d jobtracker -c "SELECT agent_name, status FROM agent_runs;"
```
